# Definition of Done

The Definition of Done is used to decide whether a task, user story, or feature is truly complete. A task should not be moved to “Done” unless it meets the criteria below.

---

## Did We Build the Right Thing?

- The completed work fulfills the needs of the assigned user story.
- The work matches the related acceptance criteria.
- The Product Owner has checked the feature from a user/product perspective.
- The feature supports the overall goal of Wheel Time and does not add unnecessary scope.
- Any known limitations or unfinished edge cases are documented.

---

## Does It Run Correctly?

- The code compiles successfully.
- The project builds without introducing new errors.
- The feature works when tested locally.
- The feature does not break existing launcher behavior.
- The work passes the relevant automated tests, if tests exist.
- The work has been manually tested for the expected user flow.

---

## Is the Work in the Repository?

- The code is pushed to the GitHub repository.
- The work is committed with a clear commit message.
- Any related documentation, screenshots, or notes are also pushed.
- The branch or pull request is connected to the related task when possible.
- Build files, temporary files, and personal IDE files are not included unless needed.

---

## Has the Work Been Reviewed?

- At least one teammate has reviewed the work or tested the feature.
- The Product Owner has checked that the feature matches the intended product behavior.
- Documentation is clear enough that another teammate can understand the change.
- Any feedback from review has been addressed or documented for later.

---

## Documentation Requirements

- Any new feature or behavior is documented in the README, sprint document, or relevant project notes.
- Any new setup/build step is documented.
- Any known bug, shortcut, or missing edge case is written down.
- Code comments are added only where they help explain unclear logic.

---

## Final Checklist

Before marking a task as done, confirm:

- [ ] The work matches the user story and acceptance criteria.
- [ ] The code compiles.
- [ ] The feature was tested.
- [ ] Relevant documentation was updated.
- [ ] The code was pushed to GitHub.
- [ ] A teammate or Product Owner reviewed the result.
- [ ] Known issues are documented.
