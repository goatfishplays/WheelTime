## **Actions to Stop Doing**

### **Stop waiting too long to claim tasks**

The team should stop leaving tasks unassigned until the end of the sprint. This made it harder to tell what each person was responsible for and reduced the amount of time available to complete Sprint 1 work.

### **Stop keeping tasks too vague**

The team should stop using broad tasks such as “research Windows API” without breaking them into smaller, more specific tasks. Smaller tasks are easier to estimate, assign, and move across the scrum board.

## **Actions to Start Doing**

### **Start assigning clear owners for each task**

The team should assign each task to a specific team member at the start of the sprint. This will make responsibilities clearer and help prevent duplicated or missed work.

### **Start updating the scrum board more often**

The team should update the scrum board whenever a task moves from not started, to in progress, to completed. This will make progress more visible and help the Scrum Master track the sprint.

### **Start writing short research notes in the repo**

For spike tasks, the team should add short notes or documentation to the repository. This makes research useful to the whole team instead of only being known by the person who did it.

### **Start making tasks small enough to complete in one work session**

The team should aim for tasks that are around 1–6 ideal hours. This matches the sprint planning format and makes it easier to finish and track work.

## **Actions to Keep Doing**

### **Keep focusing on the Windows MVP first**

The team should continue focusing on the main Windows MVP before attempting stretch goals. This keeps the project realistic and aligned with the core goal of building a real-time launcher.

### **Keep separating stretch goals from required sprint work**

The team should continue keeping Linux support, Discord/OBS integrations, custom themes, and other advanced features in the stretch/backlog section. This avoids overcommitting during early sprints.

### **Keep using user stories to organize work**

The team should continue organizing tasks around user stories because it connects development work to the actual user value of the product.
